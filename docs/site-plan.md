# aSaaSinators Website Plan

## Goal

Create a no-build GitHub Pages landing page that explains a graceful SaaS exit strategy powered by agents.

## Audience

Operators, founders, finance leaders, and technical teams who feel trapped by seat sprawl, generic SaaS workflows, and migration risk.

## Primary message

**The best SaaS is no SaaS.**

The page should immediately say that aSaaSinators helps teams get there without a cliff: the old SaaS keeps running while the agent becomes the front door for work, compresses human seats, mirrors data, rebuilds useful components, and eventually makes the SaaS app optional.

## Required narrative order

1. **Hero:** Make the paradigm claim and explain the no-cliff migration promise.
2. **Proof strip:** Lead with customer benefits, not the internal “one seat” tactic.
3. **Thesis:** Explain why agents changed the SaaS cost curve and why generic SaaS UI is now a liability.
4. **Human-glue / integration section:** Show that companies use people to bridge many tools and that agents collapse those handoffs.
5. **Process:** Show the single-seat entry tactic in context: seat the agent, make it the front door, compress seats, replace data calls.
6. **Operating model:** Explain requests, facilitation, data model, and component replacement.
7. **Graphs:** Show the two gates: user queries route through agent, then SaaS data calls go to zero.
8. **Access ladder:** Explain API/export/browser fallback.
9. **Calculator:** Keep the illustrative seat-compression model as a pressure gauge, not a quote.
10. **Offer:** Explain first app, known app, and multi-app consolidation.
11. **Closing:** Reinforce invisible cancellation.

## Visual approach

- Dark premium B2B landing page.
- Neon lime/magenta against black/navy.
- Generated artwork should be text-free, logo-free, and abstract.
- Graphs should be legible at desktop/tablet/mobile sizes.
- No build step; static HTML/CSS/JS only.

## Copy constraints

- Do not use “one extra seat” as a top-line benefit.
- The one-seat concept belongs in the process section and graphs.
- Prefer “front door for work,” “request conduit,” “facilitation layer,” and “compatibility layer.”
- Do not imply the service replaces employees. It facilitates their requests and adapts to their needs.
- Explicitly mention the cutoff logic: user queries to the agent first; SaaS data calls eventually reach zero.

## Site acceptance checks

- Local validator passes.
- No build tooling appears in repo or Pages workflow.
- Desktop, tablet, mobile, and narrow-mobile screenshots pass visual QA.
- Full-page desktop/tablet/mobile screenshots pass visual QA.
- Live Pages URL is cache-busted and verified after deployment.
- Live assets match local committed assets.
